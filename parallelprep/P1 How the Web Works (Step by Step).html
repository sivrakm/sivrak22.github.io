üåç How the Web Works (Step by Step)

üåê What happens when I go to google.com in my browser?

You might wonder — is the website already stored on your device?

No, it's not.

üí° So where do websites actually come from?

When you type google.com and press Enter, your browser sends a request over the internet to a server.

A server is just a computer — but it's always on, connected to the internet, and configured to respond to web requests.

Once the server receives your request, it sends back the necessary files for the web page, such as:

  • HTML (structure)
  • CSS (styling)
  • JavaScript (interactivity)
  • Images, fonts, and other assets

Your browser then renders these files into the interactive web page you see.

üñ•Ô∏è What is a Server in Practice?

You don’t need to own a physical machine to run a server. There are companies that sell or rent servers — also known as cloud providers.

Some popular cloud providers are:

  • Amazon Web Services (AWS)
  • DigitalOcean
  • Google Cloud Platform (GCP)
  • Microsoft Azure

These companies give you access to virtual machines (VMs) — which are just computers running somewhere in a data center, connected to the internet 24/7.

You can upload your website to one of these VMs (servers), and the whole world can access it through a browser.

üåç What Really Happens: From Browser to Server

The browser (your computer) sends a request to the server using the domain name — for example: google.com or example.com.

But wait — how does the browser know where that domain name points to?

üß≠ DNS (Domain Name System)

Think of DNS like the phone book of the internet.

  • Just like a phone book maps names to phone numbers,
  • DNS maps domain names (like google.com) to IP addresses (like 142.250.195.14).

Here's what happens behind the scenes:

  1. You type google.com in your browser.
  2. Your browser asks the DNS: “Hey, what’s the IP address of google.com?”
  3. DNS replies with the actual IP address of Google's server.
  4. Now your browser knows where to send the request — to that specific IP address.
  5. The server at that IP responds with the website files.
  6. Your browser renders the web page.

üí° In Simple Terms:

  • Domain Name = Human-friendly (like google.com)
  • IP Address = Computer-friendly (like 142.250.195.14)
  • DNS = The translator between them

Without DNS, we’d have to memorize numbers to visit websites — just like calling friends by their phone numbers instead of their names!

üîÑ Understanding Request & Response Using Browser DevTools

When your browser communicates with a website, it goes through a Request → Response cycle.

Let’s see what that means in action using your browser's DevTools.

üõ†Ô∏è Step-by-Step: Using DevTools to See the Request/Response

  1. Open your browser (Chrome or Edge works well).
  2. Press F12 or right-click anywhere on the page and select Inspect.
  3. Click on the Network tab.
  4. Now refresh the page (Ctrl + R or Cmd + R).

You'll see a bunch of rows appear — each row represents a request the browser made.

üîç What to Look For

Click on the first row (usually the main HTML document). Then look at these tabs:

‚ñ∂Ô∏è Headers Tab:

This shows both Request Headers and Response Headers.

  • Request Headers: What the browser sent

    • Example: GET / HTTP/1.1
    • Host, User-Agent, Accept-Type, etc.

  • Response Headers: What the server sent back

    • Example: HTTP/1.1 200 OK
    • Content-Type, Server, Cache-Control, etc.

‚ñ∂Ô∏è Preview or Response Tab:

  • This shows the actual content returned by the server — usually HTML or JSON.

‚öôÔ∏è Key Terms

  • Request (req): Your browser asking for a file (HTML, CSS, JS, etc.)
  • Response (res): The server sending back that file

üì¶ What Happens After the HTML is Loaded?

The initial response from the server usually contains just the HTML file.

But inside that HTML, there are references to other resources, like:

  • Images (<img src="logo.png">)
  • CSS stylesheets (<link rel="stylesheet" href="style.css">)
  • JavaScript files (<script src="main.js">)
  • Fonts, icons, videos, etc.

üîÅ How the Browser Handles These

  1. üî•The browser parses the HTMLüî•
  2. üö®It finds other linked resourcesüö®
  3. ‚ôªÔ∏èFor each resource, it sends additional GET requests to the server (or other servers)‚ôªÔ∏è
  4. üìùThese requests show up as separate rows in the Network tabüìù
  5. üåÄThe process repeats until all required files are downloadedüåÄ

Only after that does the browser render the full page as you see it — with layout, images, styles, and interactivity.

üß† The number of requests can vary depending on the complexity of the web page.
‚úÖ You can observe all of these in DevTools → Network → Filter by type (e.g., Img, JS, CSS).
